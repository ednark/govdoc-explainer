OMB M-08-23: Securing Federal DNS Infrastructure with DNSSEC




    Raw Data

    Source Text



    Summary of Primary Standards, Guidelines, and Requirements

    Security:

    1. Deploy Domain Name System Security (DNSSEC)
      • Action: Implement DNSSEC on all federal information systems by December 2009.
    2. Sign Top Level .gov Domain
      • Action: DNSSEC-sign the top-level .gov domain by January 2009.
    3. DNSSEC Controls Adoption
      • Action: Follow DNSSEC controls in NIST Special Publication 800-53r1.
    4. Enumerate .gov Domains
      • Action: List all second-level .gov domains managed by the agency.
    5. Identify DNS Service Sources
      • Action: Specify if DNS services are in-house, outsourced, or provided by another agency.
    6. Describe DNS Infrastructure
      • Action: Detail the vendors and server implementations used.
    7. Address Deployment Barriers
      • Action: Document and plan to overcome any barriers to DNSSEC deployment.
    8. Training and Piloting
      • Action: Participate in the Secure Naming Infrastructure Pilot and related training.
    9. Plan of Action and Milestones
      • Action: Create a detailed plan of action with key activities and timelines for DNSSEC implementation.

    Accessibility:

    No specific guidelines provided in the provided document summary.

    User Experience:

    No specific guidelines provided in the provided document summary.

    Compliance:

    1. Plan Submission
      • Action: Submit draft DNSSEC deployment plan to OMB by September 5, 2008.
    2. FISMA Reporting
      • Action: Report DNSSEC deployment progress in annual FISMA reports.
    3. Final Plan Agreement
      • Action: Agree on final DNSSEC implementation plan by October 24, 2008.

    Key Takeaways:

    • Security: Implement DNSSEC for enhanced security.
    • Compliance: Follow deadlines for DNSSEC plan submission and reporting.
    • Implementation Plan: Detail all .gov domains, DNS services, infrastructure, barriers, training, and milestones.


    Single Sentence Purpose: This document mandates plans and actions for securing government websites from DNS attacks by December 2009.

    Punchline Summaries:

    • DevOps Technician: Ensure all government web domains we manage are using DNSSEC to prevent DNS-related security threats.

    • Designer: Guarantee that our applications' public interfaces comply with security standards to protect user interactions.

    • Developer: Implement DNSSEC in our applications to meet government-required security measures and protect backend processes.

    • Project Manager: Coordinate and track our team’s efforts to meet the government's DNS security standards by the year-end 2009 deadline.



    As a DevOps technician ensuring our web application complies with the government standards for DNSSEC, you should focus on the following actions:

    1. Enumerate .gov Domains

      • List all second-level domains under our agency's .gov.

    2. Identify Sources of DNS Services

      • Determine if DNS services are managed in-house, outsourced, or provided by another agency.

    3. Describe DNS Server Infrastructure

      • Document DNS software in use (e.g., BIND, NSD, Microsoft Advanced Directory).
      • Record the number of DNS servers per source.

    4. Enable DNSSEC on DNS Servers

      • Configure DNS servers to support and use DNSSEC.
      • Ensure all relevant zones are DNSSEC-signed.

    5. DNSSEC Key Management

      • Implement procedures for generating, storing, and rotating DNSSEC keys.

    6. Test DNSSEC Deployment

      • Conduct internal and external tests to validate DNSSEC configurations.

    7. Monitor and Log DNSSEC Activities

      • Set up monitoring and logging to detect DNSSEC-related issues.

    8. Address Technical Barriers

      • Identify and document any technical issues impeding DNSSEC deployment.
      • Plan and execute steps to overcome these barriers.

    9. Participate in DNSSEC Training and Pilots

      • Engage with the Secure Naming Infrastructure Pilot (SNIP) for training and testing.

    10. Plan of Action and Milestones

      • Create a detailed plan outlining key activities and milestones for full DNSSEC deployment by the given deadline.

    By focusing on these actions, you can ensure that the web application infrastructure aligns with the DNSSEC requirements specified in the government standards document.



    For a Designer focusing on Human-Centered Design, UX, and UI of a web application, the following minimal list of actions is necessary to demonstrate compliance with government standards concerning DNSSEC:

    1. Understand DNSSEC Basics: Gain a basic understanding of DNSSEC to ensure design decisions do not inhibit its function.
    2. User Communication: Ensure the user interface clearly communicates security features, such as displaying secure domain indicators.
    3. Security Alerts: Design prompts or alerts to notify users if there is a potential security issue related to DNS.
    4. Performance Considerations: Design with awareness of potential DNSSEC-related latency, ensuring minimal impact on user experience.
    5. Accessibility Alignment: Ensure that any DNSSEC-related notifications or features adhere to accessibility standards.
    6. Consistency in Design: Maintain consistent design elements to build user trust in the secure nature of the application.
    7. Testing and Feedback: Include DNSSEC-related user interactions in usability testing to gather feedback and make necessary improvements.

    By implementing these actions, the designer ensures that the user experience is aligned with DNSSEC requirements and effectively communicates security to the end-users.



    To ensure a web application built with Drupal complies with the government DNSSEC standards, a developer should focus on the following key actions:

    1. Verify .gov Sub-Domain Registration:

      • Ensure the web application domain is properly registered under a .gov sub-domain.

    2. DNSSEC Compliance Check:

      • Confirm that the DNS service provider supports DNSSEC.
      • Verify DNSSEC is enabled and properly configured for the .gov sub-domain.

    3. Secure Hosting Environment:

      • Ensure the hosting environment supports DNSSEC deployment (e.g., server software like BIND, Microsoft Advanced Directory).

    4. Update DNS Records:

      • Update DNS records in DNSSEC-compatible format. Ensure DNS records such as A, AAAA, and CNAME comply with DNSSEC.

    5. Install and Configure DNSSEC Modules:

      • Install any necessary DNSSEC modules or plugins in Drupal (e.g., modules that help with domain validation or adding DNSSEC metadata).

    6. Monitor and Maintain:

      • Regularly monitor DNSSEC status to ensure there are no issues.
      • Keep DNSSEC records up to date during any DNS changes or domain migrations.

    7. Documentation and Testing:

      • Document all implemented DNSSEC configurations and processes.
      • Test the DNSSEC enabled domains for proper configuration using tools like DNSViz or Verisign’s DNSSEC debugger.

    By following these actions, a developer can ensure that the Drupal-based web application adheres to the necessary DNSSEC standards set forth in the government document.



    For a project manager overseeing a web application project aimed at ensuring compliance with the provided Government Standards Document, the following minimal list of actions should be taken:

    1. Enumerate Domains:

      • List all second-level .gov sub-domains your project will use.

    2. Identify DNS Service Providers:

      • Specify if DNS services are in-house, outsourced, or provided by another agency.

      • Schedule team participation in USG Secure Naming Infrastructure Pilot and related training workshops.

    6. Create Plan of Action and Milestones:

      • Develop a detailed action plan with key tasks, timelines, and milestones, focusing on DNSSEC implementation for .gov domains by December 2009.

    7. Ensure DNSSEC Capability:

      • Verify that DNSSEC capabilities are in place and operational for all applicable information systems.

    8. Report Compliance:

      • Regularly review and report the current level of compliance with DNSSEC standards as outlined in NIST publications.

    9. Submit Draft Plan:

      • Prepare and send the draft plan of action and milestones to OMB by the specified deadline.

    10. Track Progress:

      • Monitor and record progress regularly in line with the annual Federal Information Security Management Act (FISMA) reporting.
